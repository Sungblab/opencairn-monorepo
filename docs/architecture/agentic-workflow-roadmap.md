# Agentic Workflow Roadmap

This document defines the product and architecture direction for making
OpenCairn agents operate on notes, files, code, imports, exports, and long
running project work through one auditable workflow model.

It is a roadmap, not an implementation log. Implementation plans should be split
by phase and should update this document only when the architecture changes.

## Goal

OpenCairn should let users ask an agent to change their workspace, then show the
work as explicit actions with previews, approvals, status, outputs, and rollback
paths.

```text
user request
-> agent intent
-> typed action or generation spec
-> API permission check
-> immediate API write or Temporal workflow
-> note / file / code project / import / export result
-> action card, preview, version, and audit trail
```

The model is not "the LLM writes directly to the database" and not "the LLM
returns large base64 files in chat." The LLM proposes structured work. OpenCairn
validates, executes, stores, and displays that work.

## Existing Foundations

These foundations already exist and should be extended instead of duplicated:

- Workspaces, projects, pages, roles, and permission helpers.
- Yjs-backed collaborative notes and note version history.
- Durable chat runs and scoped retrieval.
- `runtime.Agent` plus Temporal for long-running work.
- Ingest workflows for uploads, Drive, Notion, Markdown ZIP, literature, and
  source note creation.
- Grounded retrieval, evidence bundles, graph expansion, and verifier evals.
- `agent_files` as first-class generated project files.
- Project object actions for generated file create, update, compile, and
  OpenCairn download export.
- Synthesis export and document generation foundations.
- Canvas/code sandbox surfaces.
- Plan8 run console and agent panel UI surfaces.

## Core Principles

### Server-Owned Scope

Agents and LLMs must not supply trusted `workspaceId`, `projectId`, `pageId`, or
`userId` values. The server injects authenticated scope and rejects conflicting
scope fields in action payloads.

### Typed Actions Over Fences

Fenced JSON blocks such as `agent-file` remain compatibility bridges. New
write-capable flows should use typed action contracts or workflow job specs.

### Temporal For Long Work

Work that may take seconds, call providers, parse files, generate documents, run
tests, export to external providers, or need retries belongs in Temporal. Small
metadata changes can stay in the Hono API after permission checks.

### Project Objects Are The User Model

The implementation can use `notes`, `agent_files`, source notes, synthesis runs,
and future tables internally. The user-facing model should remain project
objects in the project explorer and tab shell.

### Approval Before Destructive Or External Effects

Destructive, irreversible, external-provider, or expensive actions should create
approval cards before execution unless the user explicitly configured automation.

### Self-Host First

OpenCairn storage, preview, download, and internal workflows must work without
Google Workspace or hosted-service credentials. Provider export is optional.

## Unified Action Model

Future agentic work should converge on a ledger of typed actions and workflow
jobs. The first implementation may reuse existing tables where practical, but
new surfaces should share the same concepts:

| Field | Purpose |
| --- | --- |
| `id` | Stable action or job ID. |
| `requestId` | Idempotency key from the caller or generated by the server. |
| `workspaceId` | Server-injected workspace scope. |
| `projectId` | Server-injected project scope when applicable. |
| `actorUserId` | User who requested or approved the action. |
| `sourceRunId` | Chat run, agent run, workflow run, or import/export run that produced the action. |
| `kind` | Action family, such as `note.create`, `file.generate`, or `code.patch`. |
| `status` | `draft`, `approval_required`, `queued`, `running`, `completed`, `failed`, `cancelled`, or `reverted`. |
| `risk` | `low`, `write`, `destructive`, `external`, or `expensive`. |
| `input` | Validated action payload or generation spec. |
| `preview` | Human-readable diff, output summary, or planned changes. |
| `result` | Created note, file, code project, import job, export URL, or error. |
| `errorCode` | Stable failure code for retry and UI copy. |
| `createdAt` / `updatedAt` | Audit timestamps. |

The ledger can start as a focused table for agent actions and generation jobs.
It should not force an early migration of all existing entities into one
physical table.

## Action Families

### Note Actions

Agents should be able to operate on collaborative notes through explicit note
actions:

- `note.create`: create a note from a prompt, source, template, import result,
  research run, or generated outline.
- `note.update`: propose and apply block-level edits to existing note content.
- `note.rename`: rename a note.
- `note.move`: move a note to another folder or project position.
- `note.delete`: soft-delete a note after approval.
- `note.restore`: restore a deleted note or restore from note version history.
- `note.comment`: add an agent-authored comment with evidence or rationale.

Implementation expectations:

- Preserve Yjs as canonical collaborative note content.
- Use note version history before and after agent writes.
- Show a preview or diff before high-risk updates.
- Keep project/page permission checks in the API.
- Avoid direct DB writes from `apps/web`.

`note.update` uses a preview/apply action flow. The agent submits a typed draft
Plate value, the API reads the current `yjs_documents` state for the note,
converts that Yjs state to a Plate snapshot, and stores a structured preview
diff on the action ledger. The preview contract does not apply the draft and
does not treat `notes.content` as canonical note content.

The apply step transforms the live Yjs document's `"content"` root rather than
using `notes.content` as a write target, rejects stale previews when the Yjs
state vector changed, captures a note version before and after the agent edit,
rebuilds Yjs-derived mirrors and wiki-link indexes from the transformed Plate
value, and preserves Plate node ids carried by the draft where possible so
comments, mentions, and future block-level history remain stable.

The current web surface renders project-scoped draft `note.update` actions in
the Agent Panel. The card shows current and proposed text summaries, diff
counts, stale-preview guidance, an apply action that calls
`POST /api/agent-actions/:id/apply`, and a reject action that transitions the
ledger row to `cancelled`. Successful apply invalidates the action, note, and
note-version query surfaces so the review card does not become a parallel
editor state.

### File And Document Actions

Generated files should stay in the existing project object flow:

- `file.create`: create small text-like files such as Markdown, JSON, CSV, HTML,
  LaTeX, code, or plain text.
- `file.update`: create a new version of a generated file.
- `file.delete`: soft-delete or archive a generated file with restore support.
- `file.compile`: compile a source file, such as LaTeX to PDF.
- `file.generate`: generate binary or heavy artifacts through a worker.
- `file.export`: export a file to OpenCairn download or an external provider.

Implementation expectations:

- Keep `agent_files` as the stored-file table for the current generation track.
- Keep inline `content` or `base64` only for small text-like files.
- Do not generate large PDF, DOCX, PPTX, XLSX, image, or binary outputs through
  LLM base64.
- Large artifacts must be created by worker code and uploaded directly to object
  storage.

### Document Generation Export

The next document-generation phase should add a worker-backed export pipeline:

```text
note / chat / research / synthesis source
-> generation spec
-> Temporal workflow
-> DOCX / PPTX / XLSX / PDF artifact
-> object storage
-> agent_files row
-> project tree, tab viewer, download, and export actions
```

Recommended first targets:

| Target | Recommended Phase 1 path |
| --- | --- |
| PDF | Reuse LaTeX/Tectonic first; add HTML/Pandoc later only if needed. |
| DOCX | Generate through `python-docx` or Pandoc from a canonical document spec. |
| PPTX | Generate through `python-pptx` from a slide spec. |
| XLSX | Generate through `openpyxl` from table and sheet specs. |

The first phase should produce reliable OpenCairn project files. Google Drive,
Docs, Sheets, and Slides export should be a later phase after the internal
generation job model is stable.

### Import Actions

Existing import flows should be reachable as agent actions:

- `import.upload`: import a user-uploaded file or archive.
- `import.markdown_zip`: import Markdown ZIP archives.
- `import.drive`: import Google Drive file IDs or folders through existing
  Google integration grants.
- `import.notion`: import Notion ZIP exports.
- `import.literature`: import papers by DOI, arXiv ID, or search result.
- `import.web`: import web pages where supported.

Implementation expectations:

- Keep long-running import work in existing import workflows.
- Every import job must transition to a terminal state on failure; it must not
  remain queued after workflow start errors.
- Results should link source notes, generated wiki notes, chunks, evidence, and
  project files.
- Follow-up actions such as summarize, graph, flashcards, and export can be
  chained from the import result.

### Export Actions

Exports should support both OpenCairn-native downloads and optional provider
handoff:

- `export.note`: export one note as Markdown, HTML, PDF, DOCX, or source JSON.
- `export.project`: export a project as Markdown ZIP or structured archive.
- `export.file`: export generated files in their native format or converted
  formats.
- `export.workspace`: later, export workspace backups through the backup and
  portability model.
- `export.provider`: upload or convert outputs to external providers.

Provider export metadata should track:

- provider name
- external object ID
- external URL
- export format
- status
- last exported timestamp
- error code

Google Workspace support should remain optional and separately consented:

- Google Drive upload for raw files.
- DOCX to Google Docs.
- XLSX to Google Sheets.
- PPTX to Google Slides.

### Code Project Actions

To move toward Replit, Lovable, and Cursor style code agents, OpenCairn needs a
code project model beyond single generated files:

- `code_project.create`: create a mini project with a file tree.
- `code_project.patch`: apply multi-file patches.
- `code_project.rename`: rename files or directories.
- `code_project.delete`: delete files or directories with approval.
- `code_project.install`: install dependencies in an approved sandbox.
- `code_project.run`: run tests, builds, lint, or preview commands.
- `code_project.package`: zip or export a generated code project.

Implementation expectations:

- Start with a stored file tree and patch model, not a full hosted IDE.
- Use object storage or a dedicated code-project store for snapshots.
- Keep generated apps linked to project objects and tab shell views.
- Preserve diff preview and rollback before destructive changes.

### Code Agent Execution Loop

The code agent loop should be explicit and bounded:

```text
generate or patch code
-> run allowed command
-> collect logs and artifacts
-> feed failure summary back to the agent
-> apply repair patch
-> repeat up to a configured limit
-> show final diff, logs, and preview
```

Implementation expectations:

- Run commands only inside an approved sandbox/runtime.
- Keep allowlists for commands and package managers.
- Store test/build/lint logs as workflow artifacts.
- Support cancellation and retry.
- Require approval for dependency installation, networked commands, destructive
  filesystem operations, and external deployment.

### Hosted Preview And App Runner

Hosted preview is a later layer on top of code projects:

- static HTML preview
- Vite or Next.js preview
- browser smoke checks
- screenshot capture
- preview URL lifecycle
- process cleanup

This should reuse the existing browser sandbox and worker safety model where it
fits. It must not introduce unrestricted server-side arbitrary code execution.

### Workflow Console

Agent Panel, Plan8 run console, project object jobs, import/export jobs, and
code agent loops should converge toward one workflow console model:

- action timeline
- approval cards
- retry, cancel, approve, reject
- generated notes, files, code projects, imports, and exports as deep links
- error summaries
- provider/export state
- run metadata and cost where available

The UI can remain split by surface at first, but events and status contracts
should converge so future consolidation is straightforward.

## Recommended Implementation Order

### Phase 0: Roadmap And Duplicate Guard

Status: this document.

- Record the unified direction in public architecture docs.
- Add feature registry ownership and duplicate guards.
- Update the public roadmap so contributors do not start parallel artifact
  stores or incompatible code-agent runtimes.

### Phase 1: Unified Agent Action Ledger

Build the shared action/job substrate:

- shared Zod contracts for action families and statuses
- API action route or service layer
- DB table for agent actions or project object jobs
- idempotency by `requestId`
- server-injected scope
- permission checks
- action status events for web consumers
- focused contract and API tests

This phase should not migrate every existing feature at once. It should make the
next phases easy to add without inventing a new status model each time.

### Phase 2: Note Agent Actions

Add the basic knowledge OS actions:

- create note
- update note with preview/diff
- apply approved note updates through Yjs state-vector guarded writes
- review and reject draft note updates from the Agent Panel
- keep Yjs-derived note mirrors, note versions, and wiki-link indexes in sync
- rename note
- move note
- soft-delete note
- restore note
- connect actions to note version history

This gives the agent safe CRUD control over the core workspace object before
moving deeper into generated code or external exports.

### Phase 3: Document Generation Export Pipeline

Add heavy generated files without LLM base64:

- generation spec contract
- Temporal workflow
- DOCX/PPTX/XLSX/PDF generation activities
- object storage upload
- `agent_files` registration
- project tree event
- status card and final open/download actions

Google provider export remains out of this phase.

### Phase 4: Import And Export Agent Actions

Wrap existing import/export features as actions:

- agent-started import jobs
- import result action cards
- OpenCairn download exports
- note/project/file export actions
- no queued-orphan failures
- chained follow-up actions from import results

This phase should reuse existing ingest/import workflows instead of adding a
parallel import runner.

### Phase 5: Code Project Workspace

Create the code workspace substrate:

- mini project file tree
- multi-file patches
- rename/delete/move
- snapshot/version model
- zip/download
- source and preview tabs

This phase makes generated code a first-class project object instead of a set of
unrelated single files.

### Phase 6: Code Agent Execution Loop

Add repair-loop behavior:

- run test/build/lint commands
- collect logs
- feed failures back to the agent
- apply repair patches
- limit iterations
- show final diff and logs

This is the first phase that makes OpenCairn feel like a code agent rather than
only a file generator.

### Phase 7: Hosted Preview

Add generated app previews:

- static preview first
- Vite/Next preview after process lifecycle is safe
- browser smoke and screenshots
- preview URL cleanup

This is the phase that starts approaching Replit/Lovable-style app building.

### Phase 8: Google Workspace Export

Implement optional provider handoff:

- Drive upload
- Docs/Sheets/Slides conversion
- external URL persistence
- provider status UI
- revocable grants and graceful no-provider fallback

This should build on Phase 3 and Phase 4 rather than becoming a separate file
artifact system.

### Phase 9: Workflow Console Unification

Converge the user-facing run surfaces:

- Agent Panel action cards
- Plan8 run console
- project object generation jobs
- import/export jobs
- code agent loops

This phase should not block earlier work, but every earlier phase should emit
events that make this consolidation possible.

## Non-Goals

- Do not replace Temporal with LangGraph or LangChain for the core runtime.
- Do not add a second chat-only artifact store.
- Do not let LLMs directly mutate trusted scope or bypass permission checks.
- Do not make Google Workspace required for self-hosted deployments.
- Do not add unrestricted server-side arbitrary code execution.
- Do not rebuild a full Office editor inside OpenCairn before provider handoff
  and preview workflows are proven.

## Verification Standard

Each implementation phase should define focused verification before claiming
completion:

- shared contract tests for action schemas
- API tests for permission and idempotency behavior
- DB tests when tables or migrations change
- worker tests for Temporal activities and workflow terminal states
- web tests for action cards, previews, approvals, and result links
- i18n parity when user-facing copy changes
- live-stack smoke for phases that require DB, Temporal, or object storage

Passing a package test suite is not enough unless the suite covers the specific
agentic action being claimed.
